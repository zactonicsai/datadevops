Apache NiFi — example grocery flows

Three ready-to-build NiFi flows wired to this platform's grocery scenario: simulate live sales into Kafka, branch customer feedback to a file, and call the what-if assistant on a schedule. Each flow lists the exact processors, their property values, and how to connect them — recreate any of them on the canvas in a few minutes.

Accessing the NiFi UI

NiFi is reachable two ways once the stack is up:

  • Through the load balancer: http://localhost:8080/nifi/ (note the trailing slash)
  • Directly on its own port (reliable fallback): http://localhost:8443/nifi/

It runs unsecured over HTTP for local use — no login. Give it a minute after docker compose up; NiFi is a JVM app and takes longer to start than the other services.

⚠ If you previously saw a blank page

That was a reverse-proxy path bug: NiFi 2.x builds its UI asset URLs from the X-ProxyContextPath header, which the proxy wasn't sending, so the page loaded but its CSS/JS 404'd. The /nifi/ route now forwards the correct X-Proxy* headers and proxies to NiFi's native base path. If the LB route still misbehaves in your environment, use the direct :8443 port — it bypasses the proxy entirely.

A 60-second NiFi primer

NiFi moves data as FlowFiles (a piece of content plus attributes) between processors (boxes that do one thing — generate, transform, route, publish). You wire processors together with connections, and each connection carries one or more relationships (success, failure, etc.). You build flows by dragging processors onto the canvas, configuring their properties, drawing connections, then starting them.

Processor = a step FlowFile = the data in motion Connection = a queue between steps Relationship = success / failure / …

Flow 1 · Simulate live sales into Kafka

This is the headline flow: NiFi generates a synthetic grocery sale every few seconds and publishes it to the grocery.sales Kafka topic — exactly what the message-api would produce. The ingest-worker then consumes it into Postgres, so you can watch a NiFi-driven event flow all the way to the dashboard.

The shape of the flow

1GenerateFlowFile  ──success──▶  ReplaceText  ──success──▶  PublishKafka


   (timer: every 5s)        (writes a sale JSON)         (topic: grocery.sales)

Creates an empty FlowFile on a timer, to drive the flow.

Processor 3 — PublishKafka

Publishes the FlowFile content to Kafka. In NiFi 2.x search the Add-Processor dialog for PublishKafka (it targets the modern Kafka API).

PropertyValueWhy
Kafka Brokerskafka1:29092,kafka2:29092,kafka3:29092The INTERNAL listeners — NiFi is on the same Docker network, so it uses the container names.
Topic Namegrocery.salesThe same topic the ingest-worker consumes.

✓ Verify it works

Start all three processors. Within seconds you should see FlowFiles passing through the connections. Then check Postgres: docker compose exec postgres psql -U grocery -d grocery -c "SELECT category, count(*) FROM sales WHERE sale_date = CURRENT_DATE GROUP BY category;" — the Produce count climbs as NiFi publishes. The dashboard's revenue chart picks it up on refresh.

Flow 2 · Consume feedback & branch by sentiment

This flow reads customer-feedback events out of Kafka and routes them: negative feedback gets written to a "needs attention" file, everything else is logged. It shows NiFi consuming from the platform and making a content-based routing decision — the kind of branching that's NiFi's bread and butter.

flow 2 — consume, route, write
1ConsumeKafka ──success──▶ EvaluateJsonPath ──matched──▶ RouteOnAttribute


   (grocery.customer_feedback)    (extract rating)        ├─negative─▶ PutFile


                                                           └─other────▶ LogAttribute

Processor 1 — ConsumeKafka

PropertyValue
Kafka Brokerskafka1:29092,kafka2:29092,kafka3:29092
Topic Name(s)grocery.customer_feedback
Group IDnifi-feedback (its own consumer group, separate from the ingest-worker's)
Offset Resetlatest
Why a separate Group ID?

Kafka delivers each message to one consumer per group. Using nifi-feedback (not the worker's grocery-ingest) means NiFi gets its own copy of every feedback event without stealing messages from the ingest pipeline — both consumers see everything.

Processor 2 — EvaluateJsonPath (pull out the rating)

SettingValue
Destinationflowfile-attribute
add property → rating$.data.rating
add property → theme$.data.theme

This copies the JSON's rating and theme into FlowFile attributes so the next processor can route on them without re-parsing the body.

Processor 3 — RouteOnAttribute (the branch)

SettingValue
Routing StrategyRoute to Property name
add property → negative${rating:le(2)}

The expression ${rating:le(2)} is true when the rating is 2 or below. Those FlowFiles leave via the negative relationship; everything else exits via unmatched.

Processors 4 & 5 — PutFile and LogAttribute

ProcessorConnect fromKey setting
PutFileRouteOnAttribute negativeDirectory /tmp/feedback-alerts — writes each low-rated review to a file.
LogAttributeRouteOnAttribute unmatchedJust logs; auto-terminate its success.
✓ Verify it works

Because the seed loads extra low-rated "freshness" complaints for Produce (the shrink scenario), starting this flow with offset earliest instead of latest will immediately route a batch of negative reviews into /tmp/feedback-alerts inside the NiFi container: docker compose exec nifi ls /tmp/feedback-alerts.

Flow 3 · Scheduled what-if report

This flow calls the platform's analytics what-if API on a timer and saves the answer — turning the interactive assistant into an automated daily briefing. It shows NiFi calling an internal HTTP service and persisting the result.

flow 3 — scheduled HTTP call
1GenerateFlowFile ──▶ ReplaceText ──▶ InvokeHTTP ──Response──▶ PutFile


   (daily / 24h)      (question JSON)   (POST analytics)        (save answer)
ProcessorSettingValue
GenerateFlowFileRun Schedule24 hours (or 60 sec to test)
ReplaceTextReplacement Value{"question":"What changed in Produce shrink and Meat margin this week, and what should we do?"}
InvokeHTTPHTTP MethodPOST
Remote URLhttp://analytics:8002/whatif
Request Content-Typeapplication/json
(auto-terminate)terminate Original, Retry, No Retry, Failure
PutFileDirectory/tmp/whatif-reports (connect from InvokeHTTP's Response)
Why analytics:8002 and not the load balancer?

NiFi is inside the Docker network, so it can reach the analytics service directly by its service name and internal port — no need to go back out through NGINX. (Going through http://lb/analytics/whatif would also work but adds a hop.)

✓ Verify it works

Set the schedule to 60 sec to test, start the flow, then: docker compose exec nifi cat /tmp/whatif-reports/* — you'll see the LLM's grounded answer (or, if Ollama is still warming up, the fact-based fallback). This is the same response the dashboard's what-if box returns.

Troubleshooting

SymptomCause & fix
Can't reach the UI / blank pageUse http://localhost:8443/nifi/ (direct port, bypasses the proxy). Give NiFi 1–2 min to start. Check docker compose logs nifi.
PublishKafka routes to failureCheck the brokers value uses the internal ports (:29092), not localhost:9092 — NiFi is on the Docker network.
ConsumeKafka gets nothingSet Offset Reset to earliest to read existing messages; latest only sees new ones. Confirm the topic name is exact.
InvokeHTTP times outIf calling /whatif, the LLM may be loading — raise the processor's Read Timeout, or expect the fact-based fallback.
Processor won't start (yellow ⚠)Hover the warning icon — it lists missing required properties or unconnected relationships to fix.
