Architecture & data flow

The platform follows a classic event-driven pipeline. Messages enter through a single API, flow through Kafka, get persisted to a database, and are read back by an analytics service that powers a dashboard. Everything sits behind one NGINX load balancer, so the browser only ever talks to one address.

There are five logical groups:

1. Messaging backbone

A three-broker Kafka cluster running in KRaft mode (no ZooKeeper) is the event bus. A one-shot kafka-setup job creates the 12 grocery topics. A separate single ZooKeeper exists only to give Apache NiFi its state coordination.

2. Application services

message-api (3 replicas) accepts text/JSON/binary messages and produces them to Kafka. ingest-worker consumes those events and writes them to PostgreSQL. analytics reads the database and computes KPIs, forecasts, charts and what-if answers. dashboard serves the UI. The lb (NGINX) routes the browser to all of them.

3. Data & storage

PostgreSQL is the system of record; MinIO stores uploaded binary files; the one-shot seed job loads ~90 days of demo data with built-in business scenarios.

4. What-if assistant

Ollama runs a local LLM (llama3.2); ChromaDB stores vector embeddings of the store's metrics so the assistant can retrieve relevant facts before answering.

5. The edge

NGINX is the single entry point on port 8080, load-balancing the three message-api replicas and routing each URL prefix to the right service.

The compose file

Everything is orchestrated by a single docker-compose.yml. Two features at the top of the file are worth understanding before the services themselves: the version declaration and a YAML anchor that shares common Kafka settings across the three brokers.

docker-compose.yml — shared Kafka anchor
10x-kafka-common: &kafka-common


An extension field (the x- prefix) holding settings common to all three Kafka brokers. &kafka-common defines a YAML anchor — a reusable block we can stamp into each broker with <<: *kafka-common, so we write the image and health check once instead of three times.


11  image: confluentinc/cp-kafka:7.6.1


12  restart: unless-stopped


13  healthcheck:


A health check lets other services wait for this one to be genuinely ready (condition: service_healthy) rather than merely started. Docker runs the test command on an interval and marks the container healthy only once it passes.


14    test: ["CMD", "bash", "-c", "kafka-broker-api-versions --bootstrap-server localhost:29092 ..."]


Probes the broker's INTERNAL listener on port 29092. We deliberately use 29092 (not the host-facing 9092/9093/9094) because all three brokers bind 29092 identically — using a per-broker host port would make the shared health check fail on two of them.


15    interval: 15s


16    start_period: 40s


Grace window: failures during the first 40 seconds don't count against the retry budget, giving the JVM-based broker time to boot before health is judged.

Kafka cluster (KRaft mode)

kafka1 · kafka2 · kafka3 confluentinc/cp-kafka:7.6.1
role
Distributed event log — the backbone every message flows through
mode
KRaft (Kafka Raft) — no ZooKeeper; brokers manage their own metadata
replicas
3 brokers, each also a controller (combined mode)
host ports
9092, 9093, 9094 (one EXTERNAL listener per broker)
internal port
29092 (inter-broker), 29093 (controller quorum)
durability
replication factor 3, min in-sync replicas 2

Kafka is an append-only log of events. Producers append messages to topics; consumers read them at their own pace. Topics are split into partitions for parallelism, and each partition is replicated across brokers for durability. This cluster runs in KRaft mode, where the brokers themselves form a Raft consensus group to manage cluster metadata — replacing the older ZooKeeper-based design.

docker-compose.yml — kafka1 (KRaft)
77  kafka1:


78    <<: *kafka-common


Merges in the shared anchor from line 10 — pulls in the image, restart policy, and health check. Anything below overrides or adds to it.


84      KAFKA_NODE_ID: 1


Every node needs a unique numeric id. This is how the Raft quorum and replica assignment refer to each broker.


85      KAFKA_PROCESS_ROLES: "broker,controller"


Combined mode: this process is both a broker (stores partition data, serves clients) and a controller (participates in the metadata Raft quorum). For a 3-node demo this is simplest; production usually splits these into dedicated roles.


86      CLUSTER_ID: "${CLUSTER_ID:-OFXVWG8FR8anxcz23e7kMw}"


The shared cluster identifier, with an env-override and a baked-in default. KRaft requires a pre-generated base64 UUID (from kafka-storage random-uuid); all three brokers must use the same value or they won't form one cluster.


87      KAFKA_CONTROLLER_QUORUM_VOTERS: "1@kafka1:29093,2@kafka2:29093,3@kafka3:29093"


The Raft voter roster. Each entry is nodeId@host:port on the controller listener. With three voters the quorum is two, so the cluster keeps making progress if any single controller is down.


90      KAFKA_LISTENERS: "CONTROLLER://0.0.0.0:29093,INTERNAL://0.0.0.0:29092,EXTERNAL://0.0.0.0:9092"


Three sockets the broker listens on, all bound to 0.0.0.0 (all interfaces inside the container): the controller quorum (29093), inter-broker traffic (29092), and the host-facing port (9092).


91      KAFKA_ADVERTISED_LISTENERS: "INTERNAL://kafka1:29092,EXTERNAL://localhost:9092"


The crux of Kafka networking: after a client connects, the broker hands back these addresses to use for actual data. Containers reach it at kafka1:29092 (Docker DNS); your laptop reaches it at localhost:9092. The controller listener isn't advertised because only the brokers use it.


96      KAFKA_MIN_INSYNC_REPLICAS: 2


Paired with the producer's acks=all, a write isn't acknowledged until 2 of the 3 replicas have it. This is the durability guarantee — one broker can die without losing acknowledged messages.


99      KAFKA_DEFAULT_REPLICATION_FACTOR: 3


101    ports: ["9092:9092"]


Publishes this broker's EXTERNAL port to the host so tools on your machine can connect to localhost:9092. kafka2→9093, kafka3→9094.
⚠ First-launch note

Because KRaft stores cluster metadata on disk keyed by CLUSTER_ID, if you ever reuse volumes from an older ZooKeeper-based run the brokers will refuse to format. Clear them with docker compose down -v before the first KRaft boot.

kafka-setup (topic creation)

kafka-setup one-shot
role
Creates the 12 grocery topics once the brokers are healthy, then exits
restart
"no" — it runs once and is done

Kafka can auto-create topics, but explicit creation lets us control partitions and replication. This short-lived container waits for the cluster, runs a shell script, and exits. It depends on all three brokers being healthy (not just started).

kafka-config/create-topics.sh
5BROKER="kafka1:29092"


Targets the INTERNAL listener of one broker. Topic creation is a cluster-wide operation, so contacting any single broker is enough — it forwards to the controller.


7TOPICS=(


8  "grocery.sales" "grocery.inventory" "grocery.vendor_delivery" ...


One topic per dataset: sales, inventory, vendor_delivery, customer_feedback, shelf_space, ad_program, marketing, discount_program, expiry, binary, text, json — 12 in total.


22for i in $(seq 1 30); do


A readiness loop: try up to 30 times (with sleeps) to reach the broker before creating topics, so a slow cluster start doesn't fail the job.


34  kafka-topics --bootstrap-server "$BROKER" --create --if-not-exists \


--if-not-exists makes the script idempotent — re-running it is harmless. Each topic is created with the partition and replication settings below.


37    --partitions 3 --replication-factor 3


3 partitions (parallel consumption) × 3 replicas (durability) — matching the 3-broker cluster.

ZooKeeper (for NiFi only)

zookeeper-nifi confluentinc/cp-zookeeper:7.6.1
role
State coordination for Apache NiFi — not used by Kafka anymore
nodes
Single node (always has quorum; no ensemble to misconfigure)
port
2181 (client)

Since Kafka moved to KRaft, ZooKeeper is no longer needed for the message bus. But NiFi still relies on ZooKeeper to coordinate its cluster state, so we keep a single dedicated node. A lone node trivially satisfies quorum, which sidesteps the multi-node election problems that plague small ensembles.

Sends the ruok ("are you OK?") four-letter command over the client port; a healthy ZooKeeper replies imok. The grep -q imok turns that into a pass/fail the orchestrator can read. This requires ruok to be in the whitelist above.

message-api

message-api ×3 FastAPI · kafka-python-ng
role
The single ingress for all messages: text, JSON, and binary uploads

3 (load-balanced by NGINX via Docker DNS)

A FastAPI app that validates an incoming message, wraps it with metadata, and produces it to the matching Kafka topic. Binary files are streamed into MinIO and only their metadata flows through Kafka (Kafka is for events, not large blobs). Each response includes served_by — the container hostname — so you can watch the load balancer rotate across the three replicas.

services/message-api/app/main.py

Imports the producer client. The package is kafka-python-ng (a maintained fork) but it keeps the original kafka import name — see the "Show more" above for why the original fails on Python 3.12.

The container's hostname. Returned in every response as served_by — this is what lets you prove the load balancer is rotating across the three replicas (each has a different hostname).

An allow-list of acceptable dataset names. A message for anything outside this set is rejected with HTTP 400 — preventing typos from silently creating junk topics.

A lazy singleton: the producer is created on first use and reused thereafter. This avoids a hard dependency on Kafka being reachable at import time, so the web app can start even if a broker is briefly slow.


48        _producer = KafkaProducer(


50            value_serializer=lambda v: json.dumps(v).encode("utf-8"),


Every message we send is a Python dict; this serializer turns it into UTF-8 JSON bytes on the way out, so consumers receive clean JSON.

Durability: wait until all in-sync replicas confirm the write before considering it done. Combined with the broker's min.insync.replicas=2, an acknowledged message survives one broker failure.

The main endpoint for text and JSON messages. FastAPI validates the body against the Message Pydantic model automatically.

ingest-worker

ingest-worker kafka-python-ng · psycopg2
role
Consumes every grocery topic and writes events to PostgreSQL
consumer group
grocery-ingest (lets you scale consumers later)
writes to
message_log (all events) + typed tables for known datasets

The other half of the pipeline. It subscribes to all 12 topics, and for every message it (a) appends the raw event to a message_log audit table, and (b) for recognized datasets like sales or feedback, inserts a typed row into the proper table. It has retry loops so it survives the database or Kafka not being ready at the instant it starts.

Wraps the Postgres connection in a retry loop (up to ~30 attempts with sleeps) so the worker waits patiently for the database during startup instead of crashing.

Joins the grocery-ingest consumer group and starts from the earliest offset, so no historical messages are skipped on a fresh start. The group id means you could run multiple workers and Kafka would split the partitions among them.

The main loop blocks waiting for messages and yields each one as it arrives — the classic Kafka consumer pattern.

Every event is recorded in message_log (dataset, type, topic, JSON payload). Recognized datasets additionally get a typed insert (e.g. a real row in sales), wrapped in try/except so a malformed message can't halt the stream.

NGINX load balancer

lb nginx:1.27-alpine
role
Single entry point on port 8080; routes URL prefixes and load-balances message-api
host port
8080 → 80

One front door for everything. NGINX routes by URL prefix to the right internal service and round-robins requests across the three message-api replicas. Because it uses Docker's embedded DNS resolver, it picks up all replicas automatically.

LocationRoutes toNotes
/dashboard:8003The UI (and its /proxy to analytics). 180s read timeout for slow what-if calls.
/api/message-api:8001Round-robined across 3 replicas via Docker DNS.
/analytics/analytics:8002Path rewritten to strip the prefix; 180s timeout.
/nifi/nifi:8080WebSocket + X-Forwarded headers for NiFi behind a path prefix.

nginx/nginx.conf
~resolver 127.0.0.11 valid=10s ipv6=off;


127.0.0.11 is Docker's built-in DNS. Re-resolving every 10s means NGINX discovers all three message-api replica IPs dynamically — without this, it would resolve once at startup and pin to a single replica.


~client_max_body_size 100M;


Raises the upload limit so binary file uploads through /api/upload aren't rejected (NGINX defaults to a tiny 1MB).


~location /api/ { set $api message-api:8001; proxy_pass http://$api; }


Using a set variable for the upstream forces NGINX to use the resolver above at request time (rather than caching one IP at boot), which is what enables the round-robin across replicas.


~location /analytics/ { rewrite ^/analytics/(.*) /$1 break; ... }


Strips the /analytics/ prefix before forwarding, so the analytics service sees clean paths like /metrics/kpi.

Request lifecycles — how the services work together

The configuration makes more sense once you trace a request end to end. Here are the three main flows.

1. Sending a message (text or JSON)

You POST to http://localhost:8080/api/messages. NGINX matches the /api/ location and round-robins to one of the three message-api replicas (you'll see different served_by hostnames). That replica validates the dataset, wraps your content in a metadata envelope, and produces it to the matching Kafka topic with acks=all. Kafka replicates it across brokers and returns the partition + offset. Meanwhile the ingest-worker, subscribed to that topic, consumes the message and inserts it into PostgreSQL (both the audit log and, for known datasets, a typed table).

2. Uploading a binary file

You POST a multipart file to /api/upload. NGINX (with its raised 100MB body limit) forwards to a message-api replica, which streams the bytes into MinIO under a unique object key and publishes only the metadata (key, filename, size) to the grocery.binary topic. The large payload never touches Kafka — the event log stays lean.

3. Asking a what-if question

The browser POSTs to the dashboard's /proxy/whatif, which forwards to analytics. Analytics retrieves the most relevant store facts from ChromaDB (or falls back to all facts if Chroma is down), builds a prompt, and asks Ollama to reason over them. The answer flows back through the proxy to the UI. The nested timeouts ensure that even a slow model degrades to a grounded fallback rather than an error.

Startup order

Compose's depends_on with health conditions sequences the boot so nothing races a dependency that isn't ready:

  1. zookeeper-nifi and the three kafka brokers start. Kafka forms its KRaft quorum among themselves.
  2. Once the brokers are healthy, kafka-setup creates the topics and exits.
  3. postgres starts and runs the schema; once healthy, the seed job loads demo data and exits.
  4. analytics waits for both postgres and the seed job to complete — so it never serves an empty database.
  5. ollama starts; ollama-pull downloads and warms the model.
  6. message-api, ingest-worker, dashboard, and finally nifi come up.
  7. lb (NGINX) starts last, once its upstreams exist, and opens port 8080.
Quick start

docker compose up -d --build → wait for ollama-pull to finish the model download → open http://localhost:8080. The dashboard already has data thanks to the auto-seed.
